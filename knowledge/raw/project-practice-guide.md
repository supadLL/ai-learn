# 用 AI Play 学什么

AI Play 不是只展示静态课程，它把几个关键 AI 应用概念串在一个桌面项目里：课程内容、本地知识库、embedding 检索、RAG 助手、用户资料导入、LLM API 配置和构建校验。

阅读这个项目时，可以把它当作一个小型 AI 学习工作台。左侧课程帮助理解概念，右侧知识助手展示检索和生成链路，Electron 主进程展示桌面应用如何处理文件、保存配置和调用外部 API。

学习目标不是记住每个文件，而是理解一个 AI 应用从资料到回答的完整路径。

## 先看应用入口

前端入口是 React 应用。主界面负责展示课程主题、学习层级、章节内容、进度和右侧知识助手。课程内容是结构化数据，包含主题、层级、章节、重点、练习和测验。

这种设计说明：学习内容不一定要写死在组件里。把内容放成数据结构，可以更容易搜索、统计进度、生成知识库 chunk，也能未来接入更多来源。

理解入口后，学习者应该能回答：界面如何知道有哪些课程，当前主题和层级如何切换，完成进度如何保存在本地。

## 再看知识库构建

内置知识库原始资料放在 knowledge/raw。构建脚本读取 Markdown，解析 frontmatter，按标题和长度切成 chunk，为每个 chunk 生成 feature-hash-256 embedding，然后输出 public/knowledge/index.json。

这个流程模拟了真实 RAG 的离线索引阶段。真实生产系统可能使用更强 embedding 模型和向量数据库，但核心步骤一样：清洗资料，切 chunk，生成向量，保存索引。

学习者可以从这里理解：知识库不是一整本书直接交给模型，而是先被处理成可检索的小片段。

## 再看运行时检索

运行时检索逻辑会加载课程内容、资料目录、书籍占位片段和 public/knowledge/index.json。用户提问后，系统生成 query embedding，再和 chunk embedding 算 cosine 相似度，返回 TopK。

项目里还加入了一些词法加分和查询扩展，用来改善轻量 feature hashing 对中文同义表达不敏感的问题。这说明检索质量不只取决于 embedding，也可以通过查询改写、关键词匹配和排序规则改进。

学习者可以用同一个问题多试几次，比如“RAG 是什么”“检索增强生成介绍”“什么时候用微调”，观察命中的 chunk 是否合理。

## 再看 RAG 助手

RAG 助手先做本地检索。如果没有配置 LLM API，它会返回基于检索片段的摘要式回答。如果配置了 LLM API，它会把问题和 TopK 片段交给 LLM，让模型生成更自然的答案。

这体现了 RAG 的两层：检索层负责找资料，生成层负责组织答案。没有 LLM 时，系统仍然能展示检索结果；有 LLM 时，答案更像对话。

学习者要注意：LLM 的回答质量依赖检索片段。如果检索错了，生成再流畅也可能错。

## 再看 Electron 主进程

Electron 主进程负责桌面能力：打开窗口、导入本地文件、解析 PDF、保存用户知识库、保存 Embedding 和 LLM 配置、通过 IPC 暴露能力给前端。

这说明浏览器前端和桌面能力是分层的。前端负责交互，主进程负责文件系统和系统 API。敏感操作，例如读取文件和保存 API Key，不应该随意放在普通网页环境里。

理解这一层后，学习者可以明白为什么同一个功能在浏览器预览和桌面 App 中能力不同：浏览器没有完整本地文件访问权限，Electron 有受控的桌面能力。

# 推荐阅读顺序

第一步，看课程数据结构，理解 AI Play 覆盖哪些主题。第二步，看 knowledge/raw，理解内置知识从哪里来。第三步，看构建脚本，理解 chunk 和 embedding 如何生成。第四步，看 localRag，理解检索如何算分。第五步，看 KnowledgeAssistant，理解用户提问如何变成检索和回答。第六步，看 Electron 主进程，理解导入资料和调用 API 如何实现。

按这个顺序读，初学者能从“这是什么应用”走到“一个 RAG 桌面应用怎样工作”。

## 可以动手做的小实验

实验一：新增一篇 Markdown 到 knowledge/raw，运行知识库构建脚本，观察 public/knowledge/index.json 的 chunk 数量变化。

实验二：修改某个标题或关键词，再搜索相关问题，观察检索结果变化。这能帮助理解标题和元数据对检索的重要性。

实验三：把同一个问题改成不同表达，例如“RAG 是什么”“检索增强生成有什么用”“知识库问答为什么要先查资料”，观察 feature hashing 的局限。

实验四：配置 LLM API，对比没有 LLM 时的检索摘要和有 LLM 时的自然语言回答。重点观察答案是否忠实引用片段。

实验五：准备 10 个问题作为小评估集，每次改知识库或检索规则后都跑一遍，记录命中情况。

## 从项目走向真实产品

真实产品可以在这个项目基础上继续增强。知识库可以从 JSON 换成 SQLite 或向量数据库。Embedding 可以从 feature hashing 换成神经 embedding。检索可以加入关键词混合检索和重排。权限可以加到 chunk 元数据里。评估可以做成自动脚本。

但不要一开始就追求复杂。AI Play 当前结构适合学习，因为它把链路放得比较直观：资料、索引、检索、回答、配置、导入。先理解这个闭环，再逐步替换更强组件。

成熟 AI 应用不是某一个模型，而是一组可靠工程流程。
