# AI 学习总地图

AI 应用学习可以分成六层：基础概念、模型使用、提示与工具、知识库检索、自动化 Agent、评估与生产化。零基础学习者最容易犯的错误是直接跳到某个热门词，例如 RAG、Agent 或微调，却没有先理解这些技术分别解决什么问题。

更稳的学习方式是先问三个问题：模型本身会什么，模型不知道什么，系统怎样约束模型。模型本身擅长语言理解、改写、总结、分类和生成。模型不知道私有资料、实时数据和没有放进上下文的信息。系统约束模型的方式包括提示词、结构化输出、工具调用、RAG、权限控制、评估集和人工确认。

学习路线不是记术语，而是建立判断力：遇到一个需求时，知道应该用普通提示、RAG、工具调用、微调、Agent，还是传统程序逻辑。

## 第一阶段：理解 LLM 是什么

LLM 是大语言模型。你可以把它理解成一个能根据上下文预测后续文本的系统。它不是数据库，也不是搜索引擎。它能把已有信息组织成清楚的答案，也能基于模式做推理，但如果上下文里没有可靠事实，它可能生成看似合理却错误的内容，这就是常说的幻觉。

初学者需要先掌握 token、上下文窗口、温度、系统提示词、用户提示词、模型版本、延迟和成本。token 是模型处理文本的基本单位，上下文窗口是模型一次能看到的全部内容。温度影响输出随机性，低温更稳定，高温更发散。模型版本会影响能力、速度、价格和输出风格。

这一阶段的目标不是训练模型，而是知道怎样调用模型、怎样描述任务、怎样判断输出是否可信。

## 第二阶段：写出可复用提示

提示词不是许愿文本，而是任务接口。一个可复用提示通常包含角色、目标、输入数据、约束、输出格式、例子和失败处理。比如“请总结这段话”是临时指令；“请根据以下通话记录输出 JSON，字段包括问题类型、用户情绪、是否需要人工介入；不要推测记录中没有的信息”更接近工程化提示。

提示工程的关键是把任务拆清楚：输入是什么，输出给谁用，什么算成功，什么不能做。如果输出要被程序继续处理，就应该尽量使用结构化输出，而不是让模型自由写一段自然语言。

这一阶段结束时，学习者应该能写出稳定的分类、抽取、总结、改写和格式转换提示，并能为每个提示准备几个测试样例。

## 第三阶段：理解 Embedding 和语义检索

Embedding 是把文本变成向量，让程序能用相似度搜索“意思接近”的内容。关键词搜索只能找字面匹配，embedding 检索可以找到表达不同但含义接近的内容。例如“怎么退款”和“钱什么时候退回来”字面不同，但语义接近。

Embedding 是 RAG 的基础。文档会先被切成 chunk，每个 chunk 生成 embedding 并存入索引。用户提问时，问题也被转成 embedding，再和知识库里的 chunk 计算相似度，找出最相关的片段。

学习这一层时，要特别关注 chunking。切得太短会丢上下文，切得太长会混入噪声。好的 chunk 应该是一个相对完整的信息单元，单独拿出来读也能理解。

## 第四阶段：用 RAG 补足模型不知道的知识

RAG 是检索增强生成。它的核心思想很简单：不要让模型凭记忆回答事实问题，而是先从知识库找资料，再把资料和问题一起交给模型，让模型基于资料回答。

RAG 适合知识经常变化、需要引用来源、资料很多、不适合直接塞进提示词的场景。企业知识库问答、产品文档助手、客服资料检索、内部制度查询都适合 RAG。

RAG 不能自动保证正确。它可能检索错、漏检、拿到过期资料，或者模型没有忠实使用资料。因此生产级 RAG 需要评估检索质量、答案忠实度、引用正确性和空结果处理。

## 第五阶段：工具调用和 Agent

工具调用让模型不只是回答，还能请求程序执行动作。模型输出结构化调用意图，程序负责校验参数、执行 API、处理错误并把结果返回给模型。执行权应该始终在应用程序手里，而不是交给模型自由操作。

Agent 是带循环的 AI 系统。它会观察当前状态，决定下一步，调用工具，读取结果，再继续判断是否完成。Agent 适合步骤不完全固定、需要根据中间结果调整计划的任务，例如资料搜集、代码修改、复杂排障和多系统协作。

不是所有任务都应该做成 Agent。如果流程固定、规则明确、结果容易用传统代码判断，那么普通程序或工作流往往更可靠。

## 第六阶段：评估与生产化

AI 应用要上线，必须有评估。没有评估集，提示词和模型版本的改动只能靠感觉判断。评估集应该包含真实用户问题、边界问题、恶意输入、空输入和高价值任务。

生产化还需要日志、监控、成本控制、超时处理、降级方案、权限控制和人工介入。尤其是 RAG 和 Agent，失败不一定发生在模型生成阶段，也可能发生在文档解析、检索、工具调用、上下文拼装或权限过滤阶段。

最终目标不是“模型回答得像人”，而是“系统能稳定、可解释、可回滚地完成任务”。

# 从浅到深的学习顺序

建议按这个顺序学习：先理解 LLM 基础，再练提示词，再学 embedding 和检索，再学 RAG，再学工具调用和 Agent，最后学评估与生产化。每个阶段都要做一个小练习，否则很容易只记住术语。

入门练习可以是：让模型总结一段资料，并输出固定 JSON。进阶练习可以是：把几篇 Markdown 切成 chunk，做本地检索。深入练习可以是：把检索结果交给 LLM，要求回答时附引用。生产化练习可以是：写 20 条评估样例，比较改动前后的结果。

如果学习者只想快速做一个能用的知识库助手，最小路径是：准备文档，切 chunk，生成 embedding，检索 TopK，把 TopK 和问题交给模型，要求模型只基于上下文回答并列出引用。

## 概念之间的关系

Prompt 负责把任务说清楚。Embedding 负责把资料找出来。RAG 负责把资料接入回答过程。Tool calling 负责让模型调用外部能力。Agent 负责把多步动作串成循环。Eval 负责判断系统是不是变好了。

这些技术不是互相替代的。一个真实 AI 应用可能同时使用提示词、RAG、工具调用和评估集。微调也不是 RAG 的替代品：微调更适合学习稳定格式和风格，RAG 更适合补充可变化的事实知识。

## 零基础学习者的常见误区

第一个误区是把 LLM 当搜索引擎。LLM 会生成答案，但不保证事实来自可靠来源。需要事实准确时，要接入搜索、数据库、工具或 RAG。

第二个误区是把所有信息都塞进提示词。上下文窗口有限，信息越多不一定越好，噪声会干扰模型。应该筛选和组织上下文。

第三个误区是以为 Agent 越复杂越高级。复杂循环会带来成本、延迟和错误传播。先用简单工作流解决问题，再考虑 Agent。

第四个误区是没有评估就上线。AI 输出有不确定性，必须用样例集和日志来观察质量变化。
