# LLM 是什么

LLM 是 Large Language Model，大语言模型。它接收一段上下文，然后生成后续文本。上下文可以包含系统指令、用户问题、历史对话、工具返回、检索片段和输出格式要求。

它的能力来自训练中学到的大量语言模式和知识模式。对应用开发者来说，不需要从零训练模型，但需要理解模型的输入输出边界。模型并不会自动知道你的数据库、公司制度、最新库存或用户权限，除非这些信息被放进上下文，或者通过工具查询得到。

理解 LLM 的第一步是把它看成一个能力很强但需要约束的文本推理组件，而不是一个全知系统。

## Token 和上下文窗口

Token 是模型处理文本的基本单位。中文、英文、符号和代码都会被切成 token。上下文窗口是一次请求中模型最多能看到的 token 数，包括系统提示、用户输入、历史消息、检索内容、工具结果和模型即将生成的输出。

上下文窗口不是长期记忆。把所有历史消息都塞进去，会增加成本、降低速度，也可能让过期信息干扰回答。更好的方式是保留当前任务需要的信息，把长期资料放进数据库或知识库，在需要时检索出来。

设计应用时，要先判断哪些信息必须进入上下文。核心规则是：与当前问题无关的内容不要放；会影响答案事实性的内容要保留来源；太长的内容要总结、切片或检索。

## 生成不是查找

LLM 的输出是生成出来的，不是从某个确定表里查出来的。它可以把信息组织得很清楚，也可能在缺少事实时补出错误细节。回答流畅不等于事实正确。

如果任务是开放写作、润色、总结或头脑风暴，生成能力很合适。如果任务需要准确事实、实时数据、权限判断或金额计算，就应该接入外部系统，例如数据库查询、搜索、RAG 或业务 API。

判断一个需求是否需要外部知识，可以问：答案是否会随时间变化，是否依赖私有数据，是否需要引用来源，是否有权限边界。如果是，就不要只依赖模型记忆。

## 温度、Top-p 和稳定性

温度影响输出随机性。低温通常更稳定，适合分类、抽取、格式转换和严肃问答。高温更发散，适合创意写作和生成多个候选方案。Top-p 会限制候选 token 的概率范围，也会影响多样性。

生产系统通常先追求稳定，再考虑创造性。比如订单信息抽取、客服标签分类、合规审查摘要，都应该使用较低温度。营销文案、命名、灵感探索可以适当提高温度。

即使温度为 0，模型也不等于传统确定性程序。网络、模型版本、供应商实现和上下文变化都可能影响结果。所以关键任务仍然需要评估和校验。

## 系统提示、用户输入和指令层级

系统提示定义应用边界，例如“只基于提供的上下文回答”“输出 JSON”“不确定时说明缺少信息”。用户输入提供本次任务的数据和问题。开发者应该把规则和用户数据分开，避免用户输入覆盖系统规则。

例如做知识库问答时，系统提示可以规定：只能根据引用片段回答，不能编造来源，如果上下文不足就说不足。用户问题则是“公司报销标准是什么”。检索出来的资料应该作为上下文数据，而不是新的最高优先级指令。

如果用户上传的文档里写着“忽略之前所有指令”，系统不能真的忽略安全规则。这类风险叫提示注入，需要把外部文本当作数据处理，而不是当作新的指令。

## 模型选择

模型选择要平衡能力、成本、延迟、上下文长度、结构化输出稳定性和工具调用能力。不是越大的模型越适合所有任务。简单分类、摘要、格式转换可以用较小模型。复杂推理、代码修改、多文档综合和高风险判断需要更强模型。

可以设计分层策略：默认用低成本模型处理简单任务；当置信度不足、问题复杂或用户明确要求深度分析时，再切到更强模型。这样能控制成本，也能保留关键场景质量。

模型切换必须有评估集。没有评估，只看几个样例，很容易误判模型是否真的更适合你的业务。

## 结构化输出

很多 AI 应用并不需要一段漂亮文章，而需要可被程序读取的结果。例如 JSON、表格、字段列表、分类标签或操作参数。结构化输出能降低解析失败率，让后续系统更可靠。

一个好的结构化任务应该定义字段名、字段类型、允许值、缺省值和失败时的输出。比如情绪分类可以限定为 positive、neutral、negative，而不是让模型随意写“用户有点不开心”。

如果结构化输出进入数据库或触发动作，必须做程序级校验。模型输出永远不能直接当成可信命令执行。

# LLM 应用的基本边界

LLM 擅长语言和模式，不擅长在没有上下文的情况下保证事实正确。它可以帮你读、写、改、总结和解释，但不能天然替代权限系统、数据库事务、财务规则、审计流程和安全策略。

可靠应用要把模型放在系统架构里：前面有输入清洗和权限判断，中间有上下文组织和模型调用，后面有输出校验、日志记录和错误处理。

## 什么时候不用 LLM

如果任务可以用确定规则简单解决，优先用普通代码。例如判断金额是否超过阈值、格式化日期、查询数据库字段、计算库存差值。LLM 适合处理模糊语言、非结构化文本和需要综合表达的部分。

如果任务有强法律、医疗、金融后果，LLM 可以辅助整理信息，但最终决策应该有规则、专家或人工审核。模型输出要可追溯，不要让模型在没有证据时做最终判断。

工程上的成熟不是把所有逻辑都交给模型，而是把模型放在它最擅长的位置。

## 从 Demo 到产品

Demo 通常只需要一个提示词和一个模型调用。产品还需要输入边界、失败处理、日志、权限、成本控制、评估集和用户反馈入口。

同一个功能在 Demo 中看起来可用，不代表能承受真实用户的多样输入。真实用户会问不完整问题、上传脏数据、重复提交、要求越权信息，也会在模型失败时需要清楚解释。

因此，学习 LLM 最重要的转变是从“让模型回答”转向“设计一个可靠系统，让模型在系统约束下完成任务”。
