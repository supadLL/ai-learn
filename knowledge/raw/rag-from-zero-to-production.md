# RAG 解决什么问题

RAG 是 Retrieval-Augmented Generation，检索增强生成。它解决的是“模型需要基于外部资料回答”的问题。模型本身不一定知道你的内部文档、最新政策、产品手册或用户上传资料，所以系统先检索资料，再让模型基于资料回答。

RAG 的价值不是让模型变聪明，而是让回答有依据。它把事实来源从模型参数转移到可更新、可审计、可引用的知识库。

最小 RAG 链路是：文档切块，生成 embedding，存入索引；用户提问，生成 query embedding，检索 TopK；把 TopK 和问题放进提示词，让 LLM 只基于上下文回答。

## 文档解析

RAG 的第一步是把原始资料变成干净文本。资料可能来自 Markdown、PDF、网页、Word、数据库记录、客服工单或内部 Wiki。解析质量会直接影响检索质量。

PDF 可能有页眉页脚、断行、表格错位和乱码。网页可能有导航、广告和重复内容。Word 文档可能有标题层级、表格和图片说明。解析时要尽量保留标题、段落、列表、表格语义和来源信息。

如果原文解析就很差，后面 embedding 和 LLM 再强也很难救回来。

## Chunking

Chunking 是把长文档切成较小片段。好的 chunk 应该表达一个完整主题，既不太短也不太长。太短会缺少上下文，太长会混入多个主题，导致检索结果不精准。

常见策略包括按标题切、按段落切、固定长度切、带 overlap 切、按语义切。学习阶段可以先按标题和段落切，再加少量 overlap。生产阶段要根据评估结果调 chunk 大小。

每个 chunk 都应该带元数据：文档名、章节标题、来源 URL、版本、更新时间、权限标签、页码或段落位置。没有元数据，后续引用和权限过滤会很困难。

## Embedding 和向量索引

Embedding 把 chunk 变成向量。向量索引用来快速找到与用户问题相似的 chunk。检索时，用户问题也会被转成向量，再和 chunk 向量计算相似度。

Embedding 模型要和语言、领域、数据类型匹配。中文资料要确认中文效果。代码资料、法律资料、医学资料、客服资料可能需要不同评估。维度越高不一定越好，还要看成本、速度和索引大小。

同一个索引里最好不要混用不同 embedding 模型生成的向量。不同模型的向量空间不一致，直接比较会失真。如果更换 embedding 模型，通常需要重建索引。

## 检索 TopK

TopK 是返回最相关的 K 个片段。K 太小可能漏掉关键资料，K 太大可能把噪声带进上下文。常见做法是先取较多候选，再用重排模型或规则重新排序，最后放入有限数量片段。

只靠向量检索有时会漏掉精确关键词，例如产品型号、错误码、合同条款编号。生产系统常用混合检索：向量检索负责语义相似，关键词检索负责精确匹配，再合并排序。

检索结果要去重，避免同一段内容重复进入上下文。重复资料会浪费窗口，也可能放大错误证据。

# 生成回答

检索结果不是最终答案。系统需要把用户问题、检索片段、引用编号和回答规则组装成提示词，再交给 LLM。提示词要要求模型只基于资料回答，资料不足时明确说明不足。

回答最好带引用。引用可以是文档名、章节、页码、chunk 编号或链接。引用让用户能追溯来源，也方便排查系统是否用了正确资料。

如果检索不到相关内容，不要让模型自由发挥。应该返回空结果说明、建议用户换问法，或者提示需要导入更多资料。

## 忠实度

RAG 回答最重要的质量指标之一是忠实度，也就是答案是否真的由检索资料支持。一个答案语言流畅但没有资料依据，就是幻觉。

提高忠实度的方法包括：限制模型只基于上下文回答；要求逐条引用；让模型在资料不足时拒答；减少无关上下文；对答案做二次事实核查。

忠实度不等于答案完整。模型可能忠实但漏答，也可能完整但引用错。评估时要分开看。

## 权限过滤

企业 RAG 必须处理权限。用户只能检索到自己有权访问的文档。权限过滤应该发生在检索阶段或更早，而不是生成后才要求模型不要泄露。

每个 chunk 应该带权限元数据，例如部门、角色、项目、密级、租户 ID。查询时先根据用户身份过滤可访问范围，再做向量检索。这样即使模型想回答，也拿不到无权资料。

权限系统不能只靠提示词。提示词是软约束，权限过滤是硬约束。

# RAG 评估

RAG 评估要拆层：文档解析是否正确，chunk 是否合理，检索是否命中，重排是否有效，上下文是否干净，回答是否忠实，引用是否正确，用户是否满意。

最基础的检索评估是 Recall@K：正确资料是否出现在前 K 个结果里。生成评估可以看答案相关性、忠实度、完整性、引用准确性和拒答是否合理。

评估集应该来自真实问题，而不是只写理想问题。真实用户会问“这个怎么弄”“上次那个政策还算吗”“报错 603011 是什么”，这些模糊表达比教科书式问题更能暴露系统问题。

## 常见失败模式

第一种失败是解析失败。PDF 文字顺序混乱，导致 chunk 不可读。第二种失败是切块失败，答案需要跨多个 chunk 才能完整。第三种失败是检索失败，同义表达找不到资料。

第四种失败是上下文拼装失败，把无关内容放得太多。第五种失败是生成失败，模型没有遵守资料边界。第六种失败是权限失败，用户看到了不该看的资料。

定位问题时不要只看最终答案。要把每一步的输入输出打印出来：用户问题、query、检索结果、提示词、模型回答和引用。

## 从学习版到生产版

学习版 RAG 可以使用 Markdown 文档、本地 JSON 索引和轻量 embedding。生产版通常需要增量索引、权限过滤、混合检索、重排、日志、监控、缓存和评估流水线。

学习版关注理解链路，生产版关注稳定性和可运营。不要一开始就追求复杂架构。先做最小闭环，再根据评估结果优化。

一个实用路线是：先做本地检索原型；再接入 LLM 生成引用答案；再加入用户导入资料；再加入评估集；最后加入权限、重排、监控和增量更新。
